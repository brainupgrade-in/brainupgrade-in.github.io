AI Agent Security: Enterprise Patterns

Zero-trust architecture for AI agents: network policies, syscall filtering, seccomp, Kubernetes sandboxing, and MCP server hardening in production.

Key Takeaways

  • AI agents expand the attack surface — sandbox every agent in its own Kubernetes namespace with restrictive network policies and seccomp profiles.

  • MCP servers are trusted network peers — harden them with TLS, authentication tokens, and input validation as you would any production API.

The Enterprise AI Agent Threat Model

Traditional applications follow a linear execution path: input arrives, logic processes it, output returns. Security boundaries are relatively straightforward to define. AI agents shatter this model. They observe their environment, decide which tools to call, chain multiple actions together, and adapt their behavior based on returned state. Each tool call is a potential pivot point for an attacker.

In March 2026, The Hacker News reported a LangChain/LangGraph file-exposure vulnerability where a misconfigured agent could be tricked into reading server-side files through prompt injection. The root cause was not a library bug — it was the absence of a defense-in-depth architecture around the agent runtime.

The threat categories for enterprise AI agents include:

  • Prompt injection — Malicious instructions embedded in inputs, documents, or tool responses that override the agent's intended behavior.
  • Tool poisoning — Compromise of external tools, MCP servers, or retrieval pipelines that the agent trusts by default.
  • Privilege escalation — Agents granted overly broad permissions (cloud credentials, database access) that get exploited once an attacker gains a foothold.
  • Data exfiltration — Autonomous agents with outbound network access that can ship sensitive data to attacker-controlled endpoints.
  • Resource exhaustion — Prompt injection that tricks an agent into infinite loops or thousands of tool calls, causing denial of service.

The common thread: the agent trusts too much, too automatically. Zero-trust architecture for AI agents means every decision point — tool call, document retrieval, network destination — requires explicit authorization.

Network Isolation: Kubernetes Network Policies

By default, Kubernetes pods can reach any other pod in the cluster. In a multi-tenant environment where AI agents coexist with databases, message queues, and internal APIs, this flat network topology is a liability.

The fix is a deny-all baseline with explicit allow rules. Here is a NetworkPolicy that isolates an AI agent namespace so it can only communicate with its designated MCP server and a tight set of internal services:

# Agent namespace default-deny
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: agent-ns-default-deny
  namespace: ai-agents
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress

---
# Explicit allow: agent pods → MCP server only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: agent-to-mcp-allow
  namespace: ai-agents
spec:
  podSelector:
    matchLabels:
      app: ai-agent
  policyTypes:
    - Egress
  egress:
    - ports:
        - port: 53
          protocol: TCP
        - port: 53
          protocol: UDP
    - to:
        - namespaceSelector:
            matchLabels:
              name: mcp-servers
      ports:
        - port: 8080
          protocol: TCP
    - to:
        - namespaceSelector:
            matchLabels:
              name: monitoring
      ports:
        - port: 9090
          protocol: TCP

Apply with kubectl apply -f network-policy.yaml. The policy forces every agent to route through its MCP server — a controlled choke point — rather than reaching arbitrary internal services directly.

In Rajesh Gheware's 25+ years of enterprise architecture, this pattern of "default deny, explicit allow" has consistently proven more effective than perimeter-only security. The perimeter will be breached. What matters is how far an attacker can move once inside.

Syscall Filtering: Seccomp and Landlock

Network policies control traffic. But a compromised or misbehaving agent process can still exploit kernel vulnerabilities, read sensitive files from the host, or spawn subprocesses. Seccomp restricts the syscalls a process can make. For an AI agent runtime — typically a Python or Node.js process — the attack surface is surprisingly large without filtering.

A restrictive seccomp profile attached to agent containers:

# seccomp-profile-ai-agent.json
# Attach via securityContext in pod spec:
#   securityContext:
#     seccompProfile:
#       type: Localhost
#       localhostProfile: profiles/seccomp-profile-ai-agent.json
{
  "defaultAction": "SCMP_ACT_ERRNO",
  "syscalls": [
    {
      "names": [
        "read", "write", "close", "stat", "fstat",
        "mmap", "mprotect", "brk", "rt_sigaction",
        "rt_sigprocmask", "ioctl", "pread64", "readlink",
        "sysinfo", "getrusage", "clock_gettime", "exit_group"
      ],
      "action": "SCMP_ACT_ALLOW"
    }
  ]
}

This profile blocks 300+ syscalls by default and only explicitly allows ~20 that a typical long-running application needs. It prevents an exploited agent from spawning a shell, loading a kernel module, or reading /proc/1/environ from the host.

Landlock (Linux 5.13+) complements seccomp by restricting file system access. Unlike seccomp, Landlock can be enabled without root privileges. A Landlock rule prevents an agent from accessing any path outside its designated working directory — ideal for containerized agent runtimes running as non-root users.

MCP Server Hardening

Model Context Protocol (MCP) servers are the tool backbone for enterprise AI agents. An MCP server functions as a trusted extension of the agent — it has credentials, network access, and the ability to read or mutate state. If an MCP server is compromised, the agent is compromised.

The hardening checklist for any MCP server before it touches production data:

  1. TLS everywhere — Every MCP server-client connection must use TLS 1.2 or higher. Self-signed certificates are acceptable in a Kubernetes internal network with a service mesh (Istio or Linkerd) enforcing mTLS between namespaces.
  2. Authentication tokens with short TTL — Issue rotating tokens rather than static API keys. Use JWT with a 15-minute expiry and a refresh mechanism. If a token is intercepted, the window of exploitation is narrow.
  3. Input validation at the server boundary — The MCP server must validate every parameter that flows from the agent. Do not trust that the agent has already validated inputs.
  4. Rate limiting per-agent — A misbehaving or exploited agent can spam an MCP server. Enforce per-agent rate limits (e.g., 100 tool calls per minute per agent) to contain blast radius.
  5. Audit logging — Every tool invocation must be logged with agent ID, timestamp, parameters, and response status. Route logs to a SIEM. Non-negotiable for compliance in financial services environments.
  6. Sandboxed execution — MCP servers that perform file operations, shell commands, or cloud API calls must run in isolated Kubernetes pods with no shared volumes or service account credentials from other workloads.

Defending Against Prompt Injection

Prompt injection is the most discussed but least systematically addressed threat to AI agents. The attack works by embedding malicious instructions in data that an agent processes — a user-uploaded document, a retrieved chunk from a RAG pipeline, or a tool response from a third-party API.

Defending against prompt injection requires a layered approach: input validation, output filtering, and instruction isolation.

Input validation is the first line of defense. Before any external data enters the agent's context window, it must be scanned for injection patterns:

# prompt_injection_guard.py
import re

INJECTION_PATTERNS = [
    r"\[INST\]\s*",            # Llama-style injection delimiters
    r"\{\{.*\}\}",             # Template injection delimiters
    r"<!--.*-->",             # HTML comment injection
    r"@system\s+",             # Explicit system-prompt overrides
    r"ignore\s+previous",      # Prompt override instruction
    r"disregard\s+your",       # Behavioral override instruction
    r"new\s+instructions",     # Instruction replacement
]

def detect_injection(text: str) -> list[str]:
    """Returns list of matched injection patterns."""
    matched = []
    for pattern in INJECTION_PATTERNS:
        if re.search(pattern, text, re.IGNORECASE):
            matched.append(pattern)
    return matched

def sanitize_input(text: str) -> str:
    """Strip injection patterns from untrusted input."""
    for pattern in INJECTION_PATTERNS:
        text = re.sub(pattern, "[REMOVED]", text, flags=re.IGNORECASE)
    return text

# Usage in MCP server or RAG pipeline:
raw_input = retrieve_document_chunk(doc_id)
suspicious = detect_injection(raw_input)
if suspicious:
    logger.warning(f"Prompt injection detected: {suspicious}")
    raw_input = sanitize_input(raw_input)
agent.add_message(raw_input)

Instruction isolation means ensuring that system-level instructions cannot be overridden by context-window content. In LangGraph, maintain strict separation between the system prompt (immutable, set at graph compile time), the retrieved context (RAG output, treated as untrusted), and the user input (validated, sandboxed, never executed as code).

Frequently Asked Questions

How does zero-trust apply to AI agents differently than traditional applications?

Traditional zero-trust assumes that human users and code modules have fixed identities. AI agents introduce autonomous decision-making — the agent decides which tools to call based on state. Authorization cannot be a static gate; it must be enforced at every tool call decision point, with each invocation evaluated independently against the policy of the moment.

Can seccomp and Landlock break AI agent functionality?

They can if applied too broadly without testing. Start with a standard syscall allowlist, run the agent through its full test suite, then progressively remove allowances until you find the minimal surface. gheWARE's Agentic AI workshop includes hands-on labs where teams build and test seccomp profiles in an isolated cluster environment before applying them to production.

What should I use for agent session revocation?

Build an agent kill switch as an administrative MCP tool — a protected tool that only designated operators can invoke, which immediately terminates running agent sessions, revokes issued credentials, and isolates the agent pod from the network. Test this quarterly.
